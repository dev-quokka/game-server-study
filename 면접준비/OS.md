## 프로세스 & 스레드 예상 질문

### 프로세스와 스레드가 무엇인가요?

```
프로세스는 운영체제로부터 자원을 할당받는 작업 단위이고,
스레드는 프로세스 내에서 실제로 실행되는 흐름의 단위입니다.

프로세스는 코드, 데이터, 힙, 스택 영역으로 구성된 독립적인 메모리 공간을 가지므로,
한 프로세스의 오류가 다른 프로세스에 영향을 주지 않습니다.
IPC 같은 별도 수단 없이는 프로세스간 통신이 어렵고 컨텍스트 스위칭 비용이 큽니다.

스레드는 같은 프로세스 내에서 코드, 데이터, 힙을 공유하고 스택과 레지스터만 독립적으로 가집니다.
덕분에 스레드간 통신과 전환이 가볍지만, 공유 자원에 대한 동기화 문제를 신경 써야 합니다.
```

<br>

### Critical Section을 어떻게 보호하나요?

```
임계 영역은 동기화 기법으로 보호하는데, 크게 락 기반과 락프리 기반으로 나눌 수 있습니다.

락 기반의 대표는 mutex입니다. mutex는 소유권 개념이라,
한 스레드가 락을 잡으면 풀기 전까지 다른 스레드는 진입할 수 없고 대기 큐에서 기다립니다.
락이 해제되면 OS가 대기 중인 스레드 중 하나를 깨워 락을 넘겨주는 방식으로 동기화 합니다.
mutex 외에도 짧은 임계 영역에 유리한 SpinLock, 자원 개수를 제한하는 Semaphore 같은 방법이 있습니다.

락프리 기반은 하드웨어의 원자적 연산으로 동기화합니다.
단순 증가는 fetch_add 같은 단일 명령어로 처리되고, 더 복잡한 로직은 CAS로 처리합니다.
CAS는 메모리 값이 기대했던 값과 같으면 새 값으로 바꾸고, 다르면 실패하는 연산인데,
실패 시 다시 읽어서 재시도하는 루프 형태로 임계 영역을 보호하는 방식을 사용합니다.
```

<br>

### Mutex, Semaphore, SpinLock의 차이와 각각 언제 쓰나요?

```
셋 다 동기화 기법이지만 동작 방식과 용도가 다릅니다.

Mutex는 소유권 개념의 락입니다. 한 스레드가 잡으면 다른 스레드는 대기 큐에서 잠들어 기다리고,
락이 풀리면 OS가 깨워줍니다. 가장 일반적인 동기화 방식이고,
임계 영역 길이와 상관없이 무난하게 쓸 수 있습니다.

SpinLock도 소유권 개념의 락인데, 대기 방식이 다릅니다.
락을 못 잡으면 잠들지 않고 while 루프로 계속 재시도합니다.
컨텍스트 스위칭 비용이 없는 대신 CPU를 계속 태우기 때문에, 임계 영역이 매우 짧고 멀티코어 환경일 때만 유리합니다.
단일 코어에서는 락을 못 잡은 스레드가 spin 도는 동안 락 잡은 스레드가 실행되지 못해서,
락이 풀리기까지 시간이 오래 걸리고 CPU 시간이 낭비됩니다.
결국 컨텍스트 스위칭 비용을 아끼려고 SpinLock을 썼는데, 오히려 더 비효율적인 결과가 나오게 됩니다.

Semaphore는 소유권 개념이 없고, 카운트로 자원 개수를 관리합니다.
카운트가 0이면 대기, 카운트가 증가하면 대기 중인 스레드를 깨웁니다.
소유권이 없기 때문에 락을 건 스레드가 아닌 다른 스레드가 풀어줄 수도 있어서,
DB 커넥션 풀처럼 자원 개수를 제한하고 싶을 때 적합합니다.

정리하면, 일반적인 임계 영역 보호는 mutex, 임계 영역이 매우 짧고 멀티코어면 SpinLock을,
자원 개수 제한이 필요하면 Semaphore를 사용하는것이 좋습니다.
```

### Deadlock을 발생 예시와 함께 설명해보세요

```
데드락이란 두 개이상의 쓰레드나 프로세스가
서로 소유한 자원을 요구하며 무한히 대기하는 현상입니다.

예를 들어 A 쓰레드가 1번 뮤텍스를 소유하고, B 쓰레드가 2번 뮤텍스를 소유한 상황에서,
A가 2번 뮤텍스를 B가 1번 뮤텍스를 요청하면,
둘 다 서로가 가진 락이 풀리기를 영원히 기다리게 됩니다.

데드락은 프로세스가 크래시되지 않고 그냥 멈춰있기 때문에,
단순히 처리가 오래 걸리는 건지 데드락인지 구분이 어렵다는 점이 까다롭습니다.

예방하려면 여러 락을 잡을 때 std::lock으로 여러 락을 한 번에 잡는 방법을 쓰거나,
항상 같은 순서로 잡도록 규칙을 정해 사용하는 방법이 있습니다.
```

### Atomic 연산이 뭐고 왜 빠른가요?

```
Atomic 연산은 하드웨어 차원에서 쪼개질 수 없는 원자적인 연산입니다.

일반 ++ 연산은 read-modify-write 세 단계로 나뉘어 중간에 다른 스레드가 끼어들 수 있는데,
atomic 연산은 CPU가 이 과정을 단일 명령어로 처리하기 때문에
중간에 끼어들 수 없어 원자성도 보장되는 연산입니다.
종류로는 단순 읽기/쓰기인 load/store, 카운터 증가에 쓰는 fetch_add, 값 교환의 exchange, 그리고 가장 강력한 CAS가 있습니다.

빠른 이유는 세 가지입니다. 첫째, CPU 명령어로 하드웨어가 직접 원자성을 보장하기 때문에 OS의 도움이 필요 없습니다.
둘째, 커널 모드 전환이 없어 유저 공간에서 끝납니다.
셋째, mutex처럼 잠들고 깨어나는 컨텍스트 스위칭 비용이 없습니다.

다만 atomic이 항상 빠른 건 아닙니다.
경합이 심하면 CAS 재시도 루프나 캐시 라인 경합으로 오히려 느려질 수 있어서,
단순한 연산에 적합하고 복잡한 임계 영역에는 mutex가 낫습니다.
```

<br>

### 락프리가 무엇인가요?

```
락프리는 락을 사용하지 않고 여러 스레드가 공유 자원에 안전하게 접근할 수 있도록 하는 동기화 방식입니다.

락 대신 하드웨어의 atomic 연산, 주로 CAS를 활용해서 동기화를 보장합니다.
CAS는 메모리 값이 기대했던 값과 같으면 새 값으로 바꾸고, 다르면 실패하는 원자적 연산인데,
실패하면 다시 읽어서 재시도하는 루프 형태로 자료구조를 구현합니다.

락 기반 방식과 비교하면 컨텍스트 스위칭 비용이 없고,
한 스레드가 멈추거나 느려져도 다른 스레드의 진행을 막지 않는다는 장점이 있습니다.

다만 스레드 경합이 심한 경우 CAS 재시도 루프로 CPU 자원 소모가 커지고,
이 경우 mutex 같은 락 방식보다 오히려 느려질 수 있습니다.
그래서 경합이 심하지 않고 임계 영역이 짧은 상황에서 가장 효과적이고,
실제로는 측정을 통해 락 방식과 비교해서 선택하는 것이 좋습니다.
```

<br>

### ABA 문제가 무엇인가요?

```
ABA 문제는 CAS 과정에서 발생하는 문제로,
한 스레드가 값을 비교하고 바꾸기 전에 다른 스레드가 A를 B로 바꿨다가 다시 A로 되돌려 놓으면,
처음 스레드는 값이 그대로라고 판단하고 연산을 진행해버리는 상황입니다.

단순 카운터처럼 값 자체가 의미를 갖는 경우에는 결과적으로 같은 값이라 큰 문제가 없지만,
포인터를 다룰 때는 위험합니다.
돌아온 A가 이미 해제됐다가 재할당된 메모리일 수 있어서,
잘못된 접근으로 크래시가 나거나 자료구조가 깨질 수 있습니다.

이를 해결하기 위해 포인터에 버전 카운터를 함께 묶어서 비교하는 태그드 포인터 방식을 많이 씁니다.
같은 주소라도 태그가 다르면 CAS가 실패하도록 하는 방식입니다.
이 외에도 Hazard Pointer 같은 메모리 회수 기법도 ABA를 회피하는 데 사용됩니다.
```

### 꼬리질문 대비

- #### 임계 영역이 뭐에요?
  > 임계 영역은 여러 스레드나 프로세스가 동시에 접근할 때 결과가 달라질 수 있는, 즉 race condition이 발생할 수 있는 코드 구간을 말합니다.  
  > 공유 자원에 대한 읽기·쓰기가 일어나는 부분이 대표적입니다.  
  > 이 영역은 한 번에 하나의 스레드만 들어가도록 mutex나 atomic 같은 동기화 기법으로 보호해야 합니다.

<br>

- #### race condition이 뭐에요?
  > race condition은 여러 스레드가 공유 자원에 동시에 접근할 때, 실행 순서에 따라 결과가 달라지는 현상을 말합니다.  
  > 예를 들어 두 스레드가 공유 변수를 더하는 경우, 더하기 연산은 값을 읽고, 더하고, 다시 쓰는 세 단계로 나뉩니다.  
  > 두 스레드가 동시에 읽어버리면 같은 값에 +1만을 해서 다시 쓰게 되고, 두 번 증가해야 할 값이 한 번만 증가하게 됩니다.  
  > 이처럼 실행 순서에 따라 결과가 달라지는 것이 race condition입니다.

<br>

- #### IPC가 무엇인가요?

> IPC는 Inter-Process Communication의 약자로, 메모리가 독립적인 프로세스들이 서로 데이터를 주고받을 수 있도록 운영체제가 제공하는 통신 메커니즘입니다.  
> 크게 두 가지 방식이 있습니다.  
> 첫 번째는 메시지 전달 방식으로, 파이프나 메시지 큐, 소켓처럼 운영체제가 중간에서 데이터를 전달해주는 방법입니다.  
> 두 번째는 공유 메모리 방식으로, 여러 프로세스가 같은 메모리 영역을 매핑해 직접 읽고 쓰는 방식이라 가장 빠르지만 동기화는 직접 처리해야 합니다.

<br>

- #### volatile 키워드가 멀티스레드 환경에서 동기화를 보장하나요?

> volatile은 변수를 캐시나 레지스터에 올리는 최적화를 막아서 항상 메모리에서 읽도록 강제합니다.  
> 이 덕분에 가시성은 확보되어 다른 스레드의 변경을 즉시 확인할 수 있습니다.  
> 하지만 이 과정에서 원자성은 보장하지 않기 때문에 count++ 같은 복합 연산에서는 여전히 race condition이 발생합니다.  
> 따라서 volatile은 가시성만 보장하고 원자성은 보장하지 못하기 때문에 동기화 도구로는 부족하고,  
> 원자적 연산이나 mutex 같은 별도의 동기화 장치가 필요합니다.

<br>

- #### 가시성과 원자성의 개념과 차이는 무엇인가요?

> 가시성이란 한 스레드에서 변경한 값이 다른 스레드에서도 즉시 인지할 수 있는지에 대한 속성입니다.  
> CPU 캐시나 컴파일러 최적화 때문에 한 스레드의 변경이 다른 스레드에게 바로 보이지 않을 수 있는데, 이를 해결하는 것이 가시성입니다.
> 원자성은 여러 단계로 이루어진 연산이 중간에 끼어듦 없이 하나의 작업처럼 처리되는 속성입니다.  
> 예를 들어 count++ 같은 연산은 읽기-증가-쓰기 세 단계로 나뉘는데, 이 사이에 다른 스레드가 끼어들지 못하게 보장하는 것이 원자성입니다.  
> 두 개념은 독립적이며, 멀티스레드 환경에서 race condition을 막으려면 둘 다 보장되어야 합니다.

<br>

- #### SpinLock은 언제 쓰나요?

> SpinLock은 임계 영역이 매우 짧고 멀티코어 환경일 때 사용합니다.  
> 잠들지 않고 spin 도는 방식이라 컨텍스트 스위칭 비용을 절약할 수 있는데,  
> 임계 영역이 길거나 단일 코어 환경이면 오히려 CPU만 낭비하게 되기 때문입니다.

<br>

- #### CAS 실패가 계속되면 어떻게 되나요?

> CAS는 다른 스레드가 먼저 값을 바꿨을 때 실패하고, 경합이 심하면 재시도가 반복됩니다.  
> 이때 재시도 루프로 CPU를 계속 소모해서 자원을 낭비하고, 운이 나쁜 스레드는 계속 실패해서 진전이 없는 기아 현상이 발생할 수 있습니다.  
> 그래서 경합이 심할 경우엔 mutex가 나을 수 있습니다.
