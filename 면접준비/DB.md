## RDBMS vs NoSQL 예상 질문

### RDBMS와 NoSQL이 무엇인가요?

```
RDBMS는 관계형 데이터베이스로 데이터를 테이블 형태로 저장하고
테이블 간의 관계를 정의하여 SQL을 통해 데이터를 다루는 데이터베이스입니다.

반면 NoSQL은 비관계형 데이터베이스로
RDBMS처럼 고정된 스키마나 관계 중심 구조를 사용하지 않고
유연한 스키마를 통해 다양한 형태의 데이터를 저장할 수 있는 특징을 가진 데이터베이스입니다.
```

<br>

### RDBMS와 NoSQL 각각 언제 사용하면 좋나요?

```
RDBMS는 정해진 스키마를 기반으로 데이터를 관리하고
트랜잭션을 통해 데이터 정합성을 보장하는 데 강점이 있습니다.

반면 NoSQL은 스키마가 자유롭고 수평 확장이 용이하기 때문에
대규모 트래픽이나 비정형 데이터를 처리하는 데 유리합니다.

이러한 특성에 따라 RDBMS는 데이터 간 관계가 중요하고
여러 테이블을 JOIN하여 데이터를 조회해야 하는 경우나
정합성과 안정성이 요구되는 시스템에 적합하며

NoSQL은 빠른 처리와 확장성이 중요한 서비스나
데이터 구조가 자주 변하는 환경에 적합하다고 생각합니다.
```

## Stored Procedure 예상 질문

### MSSQL과 MySQL 차이 설명해보세요

```
MSSQL과 MySQL은 모두 관계형 데이터베이스이지만
기능, 그리고 사용 환경에서 차이가 있습니다.

먼저, MSSQL은 Microsoft에서 만든 데이터베이스로
Windows 및 .NET 환경과의 연동이 강점이며
Stored Procedure나 트랜잭션 처리 기능이 매우 강력합니다.

반면 MySQL은 Oracle Corporation에서 관리하는 오픈소스 기반 데이터베이스로
Linux 환경과 웹 서비스에서 많이 사용되며
구조가 비교적 단순하고 읽기 성능과 확장성 측면에서 강점이 있습니다.

또한 MSSQL은 데이터 정합성과 안정성이 중요한 기업 시스템 에서 많이 사용되고
MySQL은 대규모 트래픽 처리나 게임 서버, 웹 서비스 환경 에서 많이 사용됩니다.

따라서 상황에 따라 적절한 데이터베이스를 선택하는 것이 중요하다고 생각합니다.
```

<br>

### DB에서 로직 처리 vs 서버에서 처리 차이는?

```
DB에서 로직을 처리하는 방식은 Stored Procedure처럼
데이터와 가까운 위치에서 로직을 수행하는 구조이고

서버에서 처리하는 방식은 애플리케이션에서 비즈니스 로직을 수행하고
DB는 데이터 저장과 조회 역할만 담당하는 구조입니다.

DB에서 처리하면 네트워크 왕복이 줄고
트랜잭션 처리나 데이터 정합성 유지에 유리한 장점이 있습니다.

반면 서버에서 처리하면 로직을 코드로 관리할 수 있어
유지보수와 확장성이 좋고, DB 종속성을 줄일 수 있습니다.

따라서 정합성이 중요하고 데이터 중심의 로직은 DB에서 처리하고
비즈니스 로직이나 자주 변경되는 로직은 서버에서 처리하는 것이 적절하다고 생각합니다.
```

<br>

### Stored Procedure는 언제 사용하는게 좋나요?

```
Stored Procedure는 SQL을 함수처럼 묶어서
데이터베이스 내부에서 실행할 수 있도록 만든 기능입니다.

주로 반복적으로 실행되는 쿼리나
여러 쿼리를 하나의 트랜잭션으로 묶어야 하는 경우에 사용합니다.

또한 Stored Procedure는 한 번 실행되면 실행 계획이 캐시되기 때문에
반복 호출 시 컴파일 비용이 줄어들어 성능상 이점이 있습니다.

그리고 애플리케이션에서 여러 쿼리를 보내는 대신
프로시저 호출 한 번으로 처리할 수 있어 네트워크 부하를 줄일 수 있고
DB 내부에서 권한을 제한할 수 있어 보안 측면에서도 장점이 있습니다.

다만 비즈니스 로직이 DB에 집중되면 유지보수가 어려워지고
확장성이 떨어질 수 있기 때문에
사용 목적에 맞게 선택하는 것이 중요하다고 생각합니다.
```

<br>

### Stored Procedure를 사용하면 네트워크 왕복 비용이 줄어드는 이유는?

```
Stored Procedure는 프로시저 이름과 파라미터만 전달하면
DB 내부에서 여러 SQL을 모두 실행하고 결과를 반환하기 때문에
여러 쿼리를 각각 요청하는 방식보다
서버와 DB 간의 네트워크 왕복 횟수를 줄일 수 있습니다.
```

<br>

### 트랜잭션만으로도 여러 쿼리를 묶을 수 있는 것 아닌가요?

```
트랜잭션은 여러 SQL을 하나의 작업으로 묶어
원자성을 보장하는 기능입니다.

반면 Stored Procedure는 해당 로직을
DB 내부에서 실행하도록 만드는 방식입니다.

따라서 트랜잭션만으로도 작업을 묶을 수 있지만
Stored Procedure를 사용하면 해당 작업을 DB 내부에서 한 번에 처리할 수 있어
네트워크 왕복을 줄이고 관리 측면에서 장점이 있습니다.
```

<br>

### Stored Procedure 자체에 트랜잭션 기능이 있나요?

```
Stored Procedure 자체는 여러 SQL을 묶어서 실행하는 기능이지만
자동으로 롤백을 보장해주는 트랜잭션 기능을 포함하고 있지는 않습니다.

따라서 데이터 정합성을 보장하려면
Stored Procedure 내부에서 트랜잭션을 명시적으로 사용해야 합니다.
```
