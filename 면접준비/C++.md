## 포인터 & 참조 예상 질문

### 포인터와 참조가 무엇인가요 ?

```
포인터는 메모리 주소를 저장하는 변수입니다.
연산자로 해당 주소의 값에 접근할 수 있고, null을 가질 수 있으며
다른 대상으로 재바인딩도 가능합니다.

참조는 기존 변수의 별칭입니다.
선언 시 반드시 초기화해야 하고, 한 번 바인딩되면 다른 대상을 가리킬 수 없습니다.
문법적으로는 원본 변수처럼 그대로 사용할 수 있어 가독성이 좋습니다.
```

<br>

### 포인터와 참조의 차이가 무엇인가요 ?

```
포인터와 참조의 차이는 크게 세 가지로 말씀드릴 수 있습니다.

첫 번째로 null 가능 여부입니다.
포인터는 null 값을 가질 수 있지만, 참조는 null을 가질 수 없고 반드시 선언 시 초기화되어야 합니다.

두 번째로 재바인딩 여부입니다.
포인터는 선언 이후에도 다른 대상을 가리키도록 변경할 수 있지만,
참조는 한 번 바인딩되면 다른 대상을 가리킬 수 없습니다.

세 번째로 문법적 사용 방식입니다.
포인터는 역참조 연산자나 주소 연산자를 명시적으로 사용해야 하는 반면,
참조는 원본 변수처럼 그대로 사용할 수 있어 코드 가독성 측면에서 유리합니다.

추가적으로 함수 인자로 전달할 때 포인터와 참조 모두 원본에 접근한다는 결과는 동일합니다.
다만 포인터는 null 체크나 동적 할당처럼 유연한 제어가 필요할 때,
참조는 반드시 유효한 객체가 보장되어야 할 때 사용하는 것이 적절하다고 생각합니다.
```

<br>

### call by value와 call by reference가 무엇인가요?

```
call by value는 값을 복사해서 전달하는 방식입니다.
함수 호출 시 전달된 값은 새로운 메모리 공간에 복사되며,
함수 내부에서는 이 복사된 값을 사용하기 때문에
원본 값에는 영향을 주지 않습니다.

반면 call by reference는 변수의 참조를 전달하는 방식으로,
함수 내부에서 원본 변수에 직접 접근하게 됩니다.
따라서 함수에서 값을 변경하면 원본에도 그대로 반영됩니다.
```

<br>

### 포인터와 참조는 언제 각각 사용하나요?

```
참조는 함수 인자로 유효한 객체가 반드시 보장될 때 사용합니다.
null이 될 수 없고 재바인딩도 안 되기 때문에 의도가 명확하고 안전합니다.

포인터는 null 가능성이 있거나, 동적 할당, 혹은 대상이 바뀔 수 있는 상황에서 사용합니다.
연결 리스트나 트리처럼 노드 간 연결을 표현할 때도 포인터가 적합합니다.

간단히 요약하면 항상 유효한 대상이 보장되면 참조, 아니면 포인터를 사용합니다.
```

<br>

<br>

### 꼬리질문 대비

- #### 포인터와 참조 성능 차이가 존재하나요?

> 성능 차이는 거의 없습니다. 참조가 내부적으로 포인터와 동일하게 구현되기 때문입니다.  
> 다만 참조는 항상 유효한 객체를 보장하기 때문에 성능보다는 안전성과 의도 표현 측면에서 구분해서 사용하는 것이 맞다고 생각합니다.

<br>

- #### const 포인터와 포인터 to const의 차이는?

> 포인터 to const는 가리키는 대상의 값을 변경할 수 없지만, 포인터가 다른 주소를 가리키는 것은 가능합니다. (포인터가 상수)  
> const 포인터는 반대로 포인터가 가리키는 주소를 변경할 수 없지만, 해당 주소의 값은 변경할 수 있습니다. (값이 상수)

## 가상함수(Virtual Function) 예상 질문

### 가상함수란 무엇인가요?

```
가상함수는 상속 구조에서 부모 클래스의 함수를 자식 클래스가 재정의할 수 있도록
virtual 키워드로 선언한 함수입니다.

호출할 함수가 런타임에 동적으로 결정되기 때문에,
부모 포인터로 자식 객체를 다룰 때 다형성을 구현할 수 있습니다.
```

<br>

### vtable(가상함수 테이블)과 vptr란?

```
가상함수를 가진 모든 클래스는 컴파일 타임에 vtable이 생성됩니다.
vtable은 해당 클래스의 가상함수 주소를 담은 함수 포인터 배열입니다.

해당 클래스의 객체가 생성될 때마다 vptr가 자동으로 세팅되어 자신의 vtable을 가리킵니다.
가상함수 호출 시 vptr가 vtable 내부의 실제 함수로 찾아가기 때문에
런타임에 올바른 함수가 호출됩니다.
```

<br>

### 가상 소멸자란 무엇이고 왜 필요한가요?

```
부모 포인터로 자식 객체를 delete할 때, 소멸자가 virtual로 선언되어 있지 않으면
부모 소멸자만 호출되고 자식 소멸자는 호출되지 않습니다.

자식에서 동적 할당한 메모리가 해제되지 않아 누수가 발생할 수 있기 때문에,
상속 구조에서는 반드시 소멸자를 virtual로 선언해야 합니다.
```

<br>

### 오버로딩과 오버라이딩의 차이는?

```
오버로딩은 함수명은 같으나 매개변수 개수, 타입, 반환형을 다르게 하여
각자 다른 함수로 정의하는 것입니다. 컴파일 타임에 실행되는 정적 바인딩입니다.

오버라이딩은 상속 구조에서 virtual로 선언된 가상 함수를 자식 객체에서 재정의하여
같은 형태의 함수를 각자 다르게 사용하는것을 말하며, 런타임에 실행되는 동적 바인딩입니다.
```

<br>

### RTTI가 무엇인가요?

```
RTTI는 Run-Time Type Information의 약자로,
런타임에 객체의 타입 정보를 확인할 수 있는 C++의 기능입니다.

대표적으로 dynamic_cast 와 typeid 두 가지가 있습니다.
dynamic_cast는 부모 포인터를 자식 타입으로 안전하게 다운캐스팅할 때 사용하고,
실패하면 nullptr를 반환합니다. typeid는 객체의 타입 이름을 런타임에 확인할 수 있습니다.

다만 RTTI는 런타임에 타입을 검사하는 비용이 있고, 가상함수 테이블에 의존하기 때문에
가상함수가 없는 클래스에서는 동작하지 않습니다.
게임 서버처럼 성능이 중요한 환경에서는 꼭 필요한 상황에서만 사용하는 것이 좋다고 생각합니다.
```

<br>

### 가상함수도 성능 오버헤드가 있나요?

```
가상함수는 호출 시 vptr가 vtable의 실제 함수를 간접 참조하기 때문에,
일반 함수 호출보다 오버헤드가 존재합니다.

또한 컴파일러가 어떤 함수가 호출될지 컴파일 타임에 알 수 없기 때문에
인라인 최적화가 불가능하다는 점도 오버헤드의 원인입니다.

다만 실제 게임 서버에서 병목이 될 정도의 오버헤드는 아니기 때문에,
성능보다는 설계 의도에 따라 사용 여부를 판단하는 것이 맞다고 생각합니다.
```

<br>

### 다중 상속시 문제점이 무엇인가요?

```
다중 상속의 가장 큰 문제는 다이아몬드 상속 문제입니다.

A를 B와 C가 각각 상속받고, D가 B와 C를 동시에 상속받으면 D 안에 A의 멤버 두개가 생기게 됩니다.
이때 D에서 A의 멤버에 접근하면 어느 경로의 멤버인지 모호성이 발생해 컴파일 에러가 납니다.

C++에서는 이를 해결하기 위해 가상 상속을 사용합니다.
B와 C가 A를 가상 상속하면 D 안에 A의 멤버가 하나만 유지됩니다.

다만 가상 상속 자체도 내부적으로 추가 포인터가 생겨 복잡도가 올라가기 때문에,
다중 상속보다는 인터페이스 분리나 컴포지션 방식으로 설계하는 것이 더 바람직하다고 생각합니다.
```

### 꼬리질문 대비

- #### 다형성이 무엇인가요?

> 다형성은 같은 인터페이스로 서로 다른 동작을 수행할 수 있는 OOP의 특성입니다.  
> 크게 두 가지로 나뉘는데, 오버로딩처럼 컴파일 타임에 결정되는 정적 다형성과,  
> 가상함수처럼 런타임에 결정되는 동적 다형성이 있습니다.

<br>

- #### 순수 가상함수가 무엇인가요?

> 순수 가상함수는 함수 뒤에 = 0을 붙여 선언하며, 자식 클래스에서 반드시 재정의하도록 강제하는 함수입니다.  
> 순수 가상함수를 하나라도 가진 클래스는 추상 클래스가 되어 직접 객체를 생성할 수 없고,  
> 인터페이스 역할을 하도록 설계할 때 사용합니다.

<br>

- #### override 키워드가 무엇인가요?

> override는 부모의 가상함수를 재정의할 때 명시적으로 표시하는 역할을 합니다.  
> override가 없어도 재정의는 되지만, 함수 시그니처가 부모와 조금이라도 다르면  
> 재정의가 아닌 새로운 함수로 인식되어 의도치 않은 동작이 발생할 수 있습니다.  
> override를 붙이면 시그니처가 맞지 않을 때 컴파일 에러로 잡아줘서 실수를 사전에 방지할 수 있습니다.

<br>

- #### dynamic_cast 실패 시 어떻게 되나요?

> dynamic_cast는 실패했을 때 대상이 포인터냐 참조냐에 따라 동작이 달라집니다.  
> 포인터에 사용했을 경우 실패하면 nullptr를 반환하기 때문에 null 체크로 안전하게 처리할 수 있습니다.  
> 반면 참조에 사용했을 경우 참조는 null을 가질 수 없기 때문에 실패 시 std::bad_cast 예외를 던집니다.  
> 따라서 참조에 dynamic_cast를 사용할 때는 반드시 try-catch로 예외 처리를 해줘야 합니다.

<br>

- #### 그럼 bad_cast가 무엇인가요?

> std::bad_cast는 dynamic_cast가 참조 타입에서 실패했을 때 발생하는 예외입니다.  
> 포인터 캐스팅은 nullptr로 조용히 실패하지만, 참조 캐스팅은 예외를 던지기 때문에 처리하지 않으면 프로그램이 종료됩니다.  
> 그래서 참조에 dynamic_cast를 사용할 때는 포인터 방식보다 더 주의가 필요합니다.

<br>

- #### 그럼 포인터와 참조 중 어떤 걸로 dynamic_cast를 쓰는게 좋은가요?

> 일반적으로 포인터 방식을 선호합니다. nullptr 반환으로 캐스팅 성공 여부를 간단하게 확인할 수 있고, 예외 처리 비용도 없기 때문입니다.  
> 참조 방식은 캐스팅 실패 자체가 프로그램 논리상 있어서는 안 되는 상황일 때, 즉 실패 시 예외로 명확하게 알려야 할 때 사용하는 것이 적합하다고 생각합니다.
