### TCP의 3-way handshake 와 4-way handshake 과정을 각각 단계별로 설명해보세요.

```
3-way handshake는 연결 수립 단계로, 양쪽이 서로의 초기 시퀀스 넘버를 교환하고 확인하는 과정입니다.
클라이언트가 SYN을 보내면 서버는 SYN와 ACK로 응답하고, 클라이언트가 ACK를 보내며 연결이 성립됩니다.

4-way handshake는 종료 단계입니다. 종료를 요청하는 쪽이 FIN을 보내면 상대는 일단 ACK만 응답하고,
남은 데이터를 모두 전송한 후에 FIN을 보냅니다. 마지막으로 요청한 쪽이 ACK를 보내며 종료됩니다.

ACK와 FIN이 분리된 이유는 TCP가 half-close를 지원해,
한쪽이 닫혀도 반대 방향으로는 아직 데이터를 보낼 수 있기 때문입니다.
```

### TCP에서 패킷 단위로 어떻게 받나요?

```
TCP는 바이트 스트림 프로토콜이라 메시지 경계를 보장하지 않기 때문에, 애플리케이션에서 직접 패킷을 조립해야 합니다.
가장 일반적인 방법은 Length-Prefix Framing으로, 패킷 헤더에 패킷 전체 크기 필드를 두는 방식입니다.
수신 측에서는 먼저 헤더 크기만큼 받아서 패킷 크기를 알아낸 뒤, 그 크기만큼 추가로 받아서 완성된 패킷을 조립합니다.

실무에서는 한 번의 recv()에 여러 패킷이 같이 들어올 수 있고 시스템 콜 비용도 있어서,
Ring Buffer에 일단 데이터를 다 쌓아두고 그 안에서 완성된 패킷을 하나씩 꺼내 처리하는 방식을 씁니다.
이때 보안상 최대 패킷 크기 검증과 엔디안 변환 처리도 필수로 들어갑니다.
```

### 꼬리질문 대비

- #### 또한 TIME_WAIT 상태가 무엇이며 왜 필요한가요?

> TIME_WAIT은 능동적으로 연결을 종료한 쪽이 마지막 ACK를 보낸 후 머무는 상태입니다.  
> 마지막 ACK가 유실되어 상대가 FIN을 재전송할 경우 다시 ACK를 보낼 수 있도록 하고,  
> 네트워크에 남아 있는 이전 연결의 지연 패킷이 동일한 IP와 포트로 생성된 새 연결에 끼어드는 것을 막기 위해 필요합니다.
