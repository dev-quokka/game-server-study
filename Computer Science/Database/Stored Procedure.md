# Stored Procedure

## Stored Procedure란?

일련의 SQL 쿼리를 하나의 함수처럼 실행하기 위한 쿼리 집합입니다.

즉, 특정 로직을 DB 내부에 저장해두고  
필요할 때 호출하여 사용하는 방식입니다.

<br>

## 특징

- 여러 SQL을 하나의 로직으로 묶어 실행 가능
- DB 내부에 저장되어 재사용 가능
- 실행 시 컴파일 및 최적화 후 캐시에 저장됨
- 매개변수 기반 호출 가능

<br>

## 동작 방식

- **최초 실행**

<img src="../Database/images/스토어드 프로시져.png" width="600" height="581"/>

<br>

<br>

1. 클라이언트가 프로시저 호출 (CALL)
2. DB에서 프로시저 검색
3. 최초 실행 시 컴파일 및 실행 계획 생성
4. 실행 계획을 캐시에 저장
5. 실행

> 실행 계획은 캐시에 저장되어 이후 재사용됨

<br>

<br>

- **재실행**

<img src="../Database/images/스토어드 프로시져 두번째 실행.png" width="510" height="380"/>

<br>

<br>

1. 캐시에 저장된 실행 계획 확인
2. 컴파일 과정 없이 실행

> 컴파일 과정이 생략되므로 실행 속도가 빨라짐

<br>

## 왜 사용할까? (장점)

### 1. SQL Server 성능 향상

- 최초 실행 시 컴파일 후 캐시에 저장
- 이후 실행은 캐시된 실행 계획 사용 (속도 향상)
- 반복 쿼리 실행 비용 감소

<br>

### 2. 네트워크 부하 감소

- SQL 전체를 보내는 대신 **프로시저 이름 + 파라미터만 전달**

<br>

- 기존 sql 방식

<img src="../Database/images/기존 sql 방식.png" width="680" height="484"/>

> 기존: SELECT + INSERT + UPDATE → 3번 호출

<br>

- stored procedure

<img src="../Database/images/스토어드 프로시져 방식.png" width="680" height="479"/>

> Stored Procedure: CALL proc → 1번 호출

<br>

### 3. 유지보수 용이

- SQL 로직을 DB에 집중 관리
- 애플리케이션 수정 없이 DB에서 변경 가능

<br>

### 4. 보안 강화

- 테이블 직접 접근 제한 가능
- SP 실행 권한만 부여 가능

<br>

## 그럼 항상 사용하는게 좋을까? (단점)

### 1. 유지보수 복잡성

- 비즈니스 로직이 DB와 애플리케이션에 분산됨
- 버전 관리 및 추적 어려움

<br>

### 2. 성능 이슈 가능

- 파라미터에 따라 실행 계획이 비효율적일 수 있음
- 필요 시 재컴파일 필요

<br>

### 3. 확장성 제한

- DB에 로직이 집중됨
- 서버 확장 및 구조 변경 어려움

<br>

## 그럼 언제 사용하는 것이 좋을까?

- 반복적으로 실행되는 쿼리가 있을 때
- 네트워크 비용을 줄이고 싶을 때
- 보안이 중요한 로직일 때

<br>

## 📌 핵심 정리

- Stored Procedure는 DB 내부에 저장된 SQL 함수
- 성능, 보안, 재사용성 측면에서 유리

하지만

- 유지보수 및 확장성 측면에서는 단점 존재
