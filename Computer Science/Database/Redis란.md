# Redis

Redis는 Key-Value 구조로 데이터를 저장하는  
오픈소스 기반의 인메모리 데이터 저장소입니다.

RDBMS와 달리 데이터를 메모리에 저장하여 빠른 처리가 가능합니다.

<br>

## Redis는 왜 필요한가?

데이터베이스는 데이터를 디스크에 저장하기 때문에  
서버 장애 시에도 데이터는 안전하게 유지됩니다.

하지만 디스크 접근 비용이 크기 때문에  
요청이 많아질수록 성능 저하가 발생할 수 있습니다.

특히 실시간 요청이 많은 게임 서버에서는  
DB 과부하가 발생할 수 있습니다.

> 이를 해결하기 위해 **캐시 서버**를 사용하며  
> 대표적인 캐시 서버가 Redis입니다.

캐시(Cache): 자주 사용하는 데이터를 임시 저장하여 빠르게 접근할 수 있도록 하는 공간

<br>

### 캐시 패턴

### Look-aside Cache

1. 클라이언트 요청
2. 서버 → Cache 조회
3. 데이터 존재 시 → 바로 반환 (Cache Hit)
4. 없으면 → DB 조회 후 Cache 저장 (Cache Miss)

### Write Back

- 모든 데이터를 Cache에 먼저 저장
- 일정 시간 후 DB에 반영

> 성능은 좋지만  
> **데이터 유실 위험이 있어 신중히 사용**

<br>

## 특징

- Key-Value 구조
- 인메모리 기반 → 빠른 속도
- 다양한 자료구조 지원  
  (String, List, Set, Sorted Set, Hash 등)

- 싱글 스레드 기반 처리  
  → 명령 처리 과정에서 Lock 없이 동작하여 성능 효율적

<br>

## 장점

#### 1. 빠른 성능

- 메모리 기반 → 디스크 접근 없음
- 평균 처리 속도 매우 빠름

#### 2. 다양한 자료구조

- 단순 캐시를 넘어  
  → 랭킹, 큐, 집합 연산 등에 활용 가능

#### 3. 기능 지원

- TTL (자동 만료)
- Pub/Sub (실시간 메시징)
- Lua Script 지원

#### 4. 복제 및 지속성

- Replication 지원
- RDB / AOF 방식으로 데이터 복구 가능

<br>

## 단점

#### 1. 데이터 유실 가능성

- 메모리 기반 특성상 장애 발생 시 데이터 유실 가능성 존재  
  → AOF, RDB 설정으로 보완 필요

#### 2. 메모리 사용량 제한

- 모든 데이터를 메모리에 저장  
  → 대용량 데이터 처리 시 비용 증가

#### 3. 관계형 데이터 처리의 한계

- JOIN과 같은 관계 연산 지원하지 않음  
  → 복잡한 관계형 데이터 처리에 비효율적

#### 4. 싱글 스레드 기반 처리

- CPU 연산이 많은 작업에는 비효율적

<br>

## 게임 서버에서의 활용

- 로그인 토큰 (JWT 저장)
- 유저 상태 정보 (온라인 여부)
- 랭킹 시스템 (Sorted Set)
- 매칭 큐
- 인벤토리 캐싱
- 세션 관리

<br>

## 📌 핵심 정리

- Redis는 **속도와 실시간 처리를 위한 인메모리 저장소**
- DB 부하를 줄이고 성능을 향상시키는 역할
- 데이터 유실 가능성이 있음으로 신중히 사용

> 게임 서버에서는  
> **캐시 + 상태 관리 + 실시간 처리**에 핵심적으로 사용됩니다.
