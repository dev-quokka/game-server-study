# RDBMS vs NOSQL

프로젝트를 개발하며 데이터베이스 선택에 대해 고민했습니다.

각 DB의 특성을 비교하며,  
**데이터의 구조와 처리 방식에 따라 적합한 DB 선택이 달라진다**는 것을 알게 되었고,  
프로젝트에 맞는 데이터베이스를 선택하는 것이 중요하다고 판단했습니다.

이제 RDBMS와 NoSQL의 차이점을 비교해보겠습니다.

<br>

## RDBMS (관계형 DB)

데이터를 테이블(행과 열) 구조로 저장하고,  
**SQL**을 사용하여 데이터 간의 관계를 관리하는 시스템입니다.

> SQL: 관계형 데이터베이스에서 사용하는 쿼리 언어

#### 저장된 데이터 예시

| id  | name   | age |
| --- | ------ | --- |
| 1   | 쿼카   | 28  |
| 2   | 웜벳   | 22  |
| 3   | 왈라비 | 26  |

<br>

### 특징

- 정해진 스키마 기반 구조
- 데이터 무결성과 트랜잭션 **ACID** 보장

> ACID
>
> - Atomicity: 전부 수행되거나 전혀 수행되지 않음
> - Consistency: 항상 일관된 상태 유지
> - Isolation: 트랜잭션 간 간섭 없음
> - Durability: 결과는 영구 저장

스키마를 반드시 준수해야 하므로 구조가 엄격함

<br>

## NoSQL (비관계형 DB)

테이블이 아닌 다양한 형태로 데이터를 저장하는 데이터베이스입니다.

고정된 스키마가 없고, 데이터 구조를 유연하게 설계할 수 있습니다.

### 특징

- 유연한 스키마 구조
- 수평 확장에 유리
- 대량 데이터 처리에 적합

## 장단점 비교

| 항목   | RDBMS              | NoSQL                   |
| ------ | ------------------ | ----------------------- |
| 구조   | 테이블 기반        | 키-값, 문서, 그래프     |
| 스키마 | 고정               | 유연                    |
| 무결성 | 강함               | 상대적으로 약함         |
| 확장성 | 수직 중심          | 수평 확장에 강함        |
| 성능   | 복잡한 쿼리에 강함 | 대량 데이터 처리에 강함 |

<br>

### 언제 어떤 DB를 선택할까?

#### RDBMS가 적합한 경우

- 데이터 관계가 중요한 경우
- 트랜잭션과 무결성이 중요한 경우
- 복잡한 JOIN이 필요한 경우

예:

- 금융 시스템
- 결제 / 트랜잭션 시스템

<br>

#### NoSQL이 적합한 경우

- 데이터 구조가 자주 변경되는 경우
- 대규모 트래픽 처리
- 빠른 읽기/쓰기 성능이 중요한 경우

예:

- 실시간 서비스
- 로그 시스템
- 게임 서버 캐시 / 상태 저장

<br>

## 📌 핵심 정리

- RDBMS는 **정확성과 관계 중심**
- NoSQL은 **확장성과 유연성 중심**

<br>

선택에 정답은 없으며,  
**데이터 특성과 서비스 구조에 따라 선택하는 것이 중요합니다**
