## 데드락 (Deadlock)

데드락이란 두 개 이상의 프로세스(또는 스레드)가 서로가 가진 자원을 기다리며  
**무한히 대기 상태에 빠지는 현상**입니다.

<br>

#### \* 데드락 발생 예시

<br>

<img src="../Operating%20System/images/데드락.jpg" width="800" height="693"/>

```
프로세스 1: 자원 1 점유 → 자원 2 대기
프로세스 2: 자원 2 점유 → 자원 1 대기

→ 서로 자원을 기다리며 무한 대기
→ Deadlock 발생
```

<br>

## 데드락 발생 조건 (4가지)

아래 4가지 조건이 **모두 만족**해야 발생

### 1. 상호 배제 (Mutual Exclusion)

- 자원은 동시에 하나의 프로세스만 사용 가능

### 2. 점유 대기 (Hold and Wait)

- 자원을 가진 상태에서 다른 자원을 기다림

### 3. 비선점 (No Preemption)

- 자원을 강제로 빼앗을 수 없음

### 4. 순환 대기 (Circular Wait)

- 프로세스들이 원형 구조로 자원을 기다림

<br>

> 위 조건중 하나라도 깨지면 데드락 발생 X

<br>

## 데드락 해결 방법

데드락을 방지하는 대표적인 방법으로  
**락 획득 순서를 항상 동일하게 유지하는 방식**이 있습니다.

예)

- 항상 A → B 순서로 lock 획득
- B → A 순서 금지

> 이는 발생 조건 중 순환 대기(Circular Wait)를 방지하는 방법입니다.

---

<br>

위와 같은 방법은 데드락을 직관적으로 방지하는 방식이며,  
아래는 이를 **이론적으로 분류한 해결 방법**입니다.

<br>

### 1. 예방 (Prevention)

4가지 조건 중 하나를 제거하여 예방

- 상호 배제 제거 → 자원 공유 (현실적으로 어려움)
- 점유 대기 제거 → 필요한 자원을 한 번에 요청
- 비선점 제거 → 자원 강제 회수
- 순환 대기 제거 → 자원 요청 순서 강제

> 단점: 성능 저하

<br>

### 2. 회피 (Avoidance)

- 항상 **Safe State 유지**
- 안전 순서(Safe Sequence)가 존재하는 경우만 자원 할당

→ 이러한 회피 전략을 구현하는 대표적인 알고리즘이 **은행원 알고리즘**입니다.

#### 은행원 알고리즘 (Banker’s Algorithm)

- 자원 할당 전에 시뮬레이션 수행
- Safe State 유지 여부 확인 후 할당

<br>

### 3. 탐지 & 회복 (Detection & Recovery)

#### 탐지

- 자원 할당 그래프(Resource Allocation Graph) 사용
- 순환 구조 발생 여부 확인

#### 회복

- 프로세스 종료
  - 전체 종료
  - 하나씩 종료

- 자원 선점
  - 자원 강제 회수 후 재할당

<br>

## 데드락이 자주 발생하는 경우

- 멀티스레드 환경
- 락(lock)을 여러 개 사용하는 경우
- 자원 획득 순서가 일관되지 않은 경우

<br>

## 📌 핵심 정리

- 데드락은 **서로 자원을 기다리며 무한 대기하는 상태**
- 발생 조건 4가지가 모두 만족해야 발생

<br>

> 예방: 조건 제거  
> 회피: 안전 상태 유지  
> 탐지/회복: 발생 후 처리

<br>

> 데드락 핵심 원인: **자원 획득 순서 문제**
