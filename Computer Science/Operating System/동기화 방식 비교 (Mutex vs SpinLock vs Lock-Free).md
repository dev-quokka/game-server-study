# 동기화 방식 비교 (Mutex vs SpinLock vs Lock-Free)

멀티스레드 환경에서는 공유 자원에 대한 접근을 제어하기 위해  
다양한 동기화 방식이 사용됩니다.

대표적인 방식으로 Mutex, SpinLock, Lock-Free가 있으며,  
각 방식은 대기 방식과 성능 특성이 서로 다릅니다.

아래에서는 각 동기화 방식의 개념과 차이를 살펴보겠습니다.

<br>

## Mutex

스레드 간 상호 배제를 위해 사용하는 가장 기본적인 락 방식입니다.

- 락을 획득하지 못하면 sleep 상태로 대기
- OS 스케줄링에 의해 스레드 전환 발생

<br>

## SpinLock

락을 획득할 때까지 계속 확인하며 대기하는 busy-wait 기반 락입니다.

- 락을 획득할 때까지 반복해서 확인 (busy-wait)
- sleep 없이 CPU를 사용하며 대기

<br>

## Lock-Free

락 없이 CAS 기반으로 동시성을 처리하는 방식입니다.

- retry 기반으로 동작 (실패 시 재시도)
- 적어도 하나의 스레드는 항상 진행 가능

<br>

## 핵심 비교

| 항목            | Mutex                                      | SpinLock                                                        | Lock-Free                                              |
| --------------- | ------------------------------------------ | --------------------------------------------------------------- | ------------------------------------------------------ |
| 잠금 대상       | 코드 영역 (Critical Section)               | 코드 영역 (Critical Section)                                    | 단일 메모리 (변수 단위)                                |
| 대기 방식       | Sleep (커널에 대기 상태로 전환, CPU 소모x) | Spin (루프 돌며 계속 확인, CPU 점유)                            | Non-Blocking (계속 작업 시도, 실패 시 재시도)          |
| 컨텍스트 스위칭 | 락 대기 시 발생 (비용 큼)                  | 없음                                                            | 없음                                                   |
| 구현 난이도     | 쉬움 (OS/라이브러리 제공)                  | 중간 (직접 구현 시 주의 필요)                                   | 매우 어려움 (ABA, 메모리 해제 등)                      |
| CPU 사용량      | 대기 중에는 거의 없음                      | 대기 중에도 계속 소모 (Busy Waiting)                            | 경합 시 높음 (계속 CAS 시도)                           |
| 데드락 가능성   | 있음                                       | 있음                                                            | 없음                                                   |
| 적합한 상황     | 락 보유 시간이 길거나, 경합이 심할 때      | 락 보유 시간이 매우 짧고, 컨텍스트 스위칭 비용을 아끼고 싶을 때 | 정밀한 성능이 필요하고, 데드락을 원천 차단하고 싶을 때 |

<br>

## Best Case & Worst Case

### 1. Mutex

- **Best Case(경합 없음)**
  > 뮤텍스 획득/해제가 빠르지만, 시스템 콜 오버헤드가 존재
- **Worst Case(경합 많음)**
  > Context Switching 비용이 급증하여 성능 저하  
  > Deadlock 발생 위험

<br>

### 2. 스핀락 (Spinlock)

- **Best Case(짧은 대기)**
  > context switching 없이 빠르게 자원 획득  
  > 매우 낮은 오버헤드
- **Worst Case(긴 대기)**
  > CPU를 계속 점유 (100%)  
  > 시스템 전체 성능 저하

<br>

### 3. 락프리 (Lock-Free)

- **Best Case(경합 적음)**
  > CAS 연산 한 번으로 처리
  > 가장 빠른 성능
- **Worst Case(경합 많음)**
  > CAS 실패 반복 (retry loop)  
  > CPU 사용 증가  
  > 하지만 시스템 전체는 멈추지 않음 (Deadlock Free)

<br>

> 결국 성능은 “경합 상황”에 따라 결정됩니다.

<br>

## 언제 무엇을 선택해야 할까?

- **Mutex**
  - 가장 기본적인 선택
  - 안정성과 유지보수 중요할 때

- **SpinLock**
  - 임계 구역이 매우 짧을 때
  - context switching 비용이 더 클 때

- **Lock-Free**
  - 고성능 / 고동시성 환경
  - 병목 구간 최적화 목적

<br>

## 📌 핵심 정리

동기화 방식은 **대기 방식과 성능 특성에 따라 선택해야 함**

<br>

> 즉, 정답이 정해진 것이 아니라  
> **상황에 맞는 동기화 방식을 선택하는 것이 가장 중요합니다**
